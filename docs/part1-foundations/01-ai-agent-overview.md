## 1. 개념 설명

### 1.1 AI 에이전트란 무엇인가?

#### 일상의 비유로 이해하기

AI 에이전트를 이해하는 가장 쉬운 방법은 **"디지털 직원"** 으로 생각하는 것입니다.

```
┌─────────────────────────────────────────────────────────┐
│                    사람(직원)의 업무 처리                   │
│                                                         │
│   👁️ 인지(Perceive)     : 이메일을 읽고, 상황을 파악한다     │
│        ↓                                                │
│   🧠 추론(Reason)       : 무엇을 해야 할지 판단한다         │
│        ↓                                                │
│   🤚 행동(Act)          : 문서를 작성하고, 회신한다          │
│        ↓                                                │
│   🔄 반복(Loop)         : 결과를 확인하고, 추가 작업한다     │
│                                                         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                  AI 에이전트의 업무 처리                    │
│                                                         │
│   📥 인지(Perceive)     : 입력 데이터를 받아 분석한다        │
│        ↓                                                │
│   🤖 추론(Reason)       : LLM이 다음 행동을 결정한다        │
│        ↓                                                │
│   🔧 행동(Act)          : 도구를 호출해 작업을 수행한다      │
│        ↓                                                │
│   🔄 반복(Loop)         : 결과를 확인하고, 필요시 반복한다   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 정의

> **AI 에이전트(AI Agent)** 란, 주어진 목표를 달성하기 위해
> **스스로 인지하고, 추론하고, 행동하는** 자율적 소프트웨어 시스템입니다.

핵심 키워드 세 가지:
1. **자율성(Autonomy)**: 사람의 매 단계 개입 없이 스스로 판단
2. **목표 지향(Goal-oriented)**: 주어진 목표를 달성하기 위해 행동
3. **도구 사용(Tool-use)**: 외부 도구(API, 데이터베이스, 파일 등)를 활용

#### 챗봇 vs 에이전트

많은 분들이 ChatGPT 같은 챗봇과 에이전트를 혼동합니다. 차이를 알아봅시다:

```
┌─────────────────────────────────────────────────────────┐
│                      챗봇 (Chatbot)                      │
│                                                         │
│   사용자: "날씨 알려줘"                                    │
│        ↓                                                │
│   챗봇: "저는 실시간 날씨를 확인할 수 없습니다."             │
│                                                         │
│   → 한 번의 질문 / 한 번의 응답 (단발성)                   │
│   → 외부 도구 사용 불가                                    │
│   → 텍스트 생성만 가능                                     │
│                                                         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                     에이전트 (Agent)                      │
│                                                         │
│   사용자: "날씨 알려줘"                                    │
│        ↓                                                │
│   에이전트: (생각) "날씨 API를 호출해야겠다"                 │
│        ↓                                                │
│   에이전트: (행동) weather_api.get("서울") 호출             │
│        ↓                                                │
│   에이전트: (응답) "서울은 현재 맑음, 22°C입니다"            │
│                                                         │
│   → 여러 단계의 추론과 행동 (연쇄적)                        │
│   → 외부 도구 사용 가능                                    │
│   → 실제 작업 수행 가능                                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 AI 에이전트 vs 전통 소프트웨어 비교

| 비교 항목 | 전통 소프트웨어 | AI 에이전트 |
|----------|---------------|------------|
| **동작 방식** | 미리 정해진 규칙(if-else) | LLM이 상황에 따라 판단 |
| **유연성** | 정해진 시나리오만 처리 | 예상하지 못한 상황도 대응 가능 |
| **입력 형태** | 구조화된 데이터 (JSON, DB) | 자연어, 비정형 데이터 포함 |
| **실행 흐름** | 고정된 순서 (A→B→C) | 동적으로 결정 (A→C or A→B→A→C) |
| **에러 처리** | 예외 코드로 사전 정의 | 상황 판단 후 대안 탐색 가능 |
| **학습** | 코드 수정 필요 | 프롬프트 변경으로 행동 조정 |
| **비용** | 서버 비용만 발생 | 서버 + LLM API 비용 발생 |
| **속도** | 매우 빠름 (밀리초) | 상대적으로 느림 (초 단위) |
| **정확성** | 100% 결정적(Deterministic) | 확률적(Probabilistic) |

> **핵심 포인트**: AI 에이전트는 "유연하지만 느리고 비용이 드는" 특성을 가집니다.
> 따라서 모든 것을 에이전트로 만들 필요는 없습니다. **에이전트가 적합한 문제**를
> 선별하는 것이 중요합니다.

#### 에이전트가 적합한 경우

- 입력이 비정형(자연어, 다양한 형식)인 경우
- 처리 과정에 판단이 필요한 경우
- 다양한 도구를 상황에 따라 선택해야 하는 경우
- 사람이 수동으로 반복하던 작업을 자동화하려는 경우

#### 에이전트가 부적합한 경우

- 단순 반복 작업 (예: 파일 복사, 정해진 형식 변환)
- 100% 정확성이 요구되는 경우 (예: 금융 거래)
- 실시간 응답이 필수인 경우 (예: 밀리초 단위 API)
- 비용이 극도로 중요한 경우

### 1.3 LLM(대규모 언어 모델) 기초

#### LLM이란?

> **LLM(Large Language Model)** 은 대규모 텍스트 데이터를 학습하여
> **다음에 올 단어를 예측**하는 AI 모델입니다.

```
┌─────────────────────────────────────────────────────────┐
│                    LLM의 동작 원리                        │
│                                                         │
│   프롬프트(입력)          LLM 모델          응답(출력)      │
│                                                         │
│   "한국의 수도는"  ───▶  [신경망 처리]  ───▶  "서울입니다"   │
│                                                         │
│   내부적으로는 "한국의 수도는" 다음에 올 가장 적절한          │
│   단어들을 확률적으로 예측하여 생성합니다.                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 프롬프트(Prompt)와 응답(Response)

LLM과의 대화는 항상 **프롬프트 → 응답** 구조입니다:

```
프롬프트 (사람이 작성)          응답 (LLM이 생성)
─────────────────          ─────────────────
"이 코드의 버그를 찾아줘:     "3번째 줄에서 인덱스
 x = list[5]               범위를 벗어날 수 있습니다.
 print(x[10])"             list에 11개 이상의 요소가
                            있는지 확인하세요."
```

#### 토큰(Token)이란?

LLM은 텍스트를 **토큰(Token)** 이라는 단위로 처리합니다.

```
┌─────────────────────────────────────────────────────────┐
│                    토큰 분해 예시                          │
│                                                         │
│   영어:  "Hello, world!"                                 │
│          → ["Hello", ",", " world", "!"]  (4 토큰)       │
│                                                         │
│   한국어: "안녕하세요"                                      │
│          → ["안녕", "하", "세요"]  (3 토큰, 모델마다 다름)   │
│                                                         │
│   코드:  "print('hi')"                                   │
│          → ["print", "('", "hi", "')"]  (4 토큰)         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

> **왜 토큰이 중요한가요?**
> - LLM API는 **토큰 수에 따라 비용**이 부과됩니다
> - 한 번에 처리할 수 있는 토큰 수에 **제한(Context Window)** 이 있습니다
> - 한국어는 영어보다 같은 의미를 표현하는 데 더 많은 토큰을 소비하는 경향이 있습니다

#### Temperature (온도)

Temperature는 LLM 응답의 **창의성(무작위성)** 을 조절하는 파라미터입니다.

```
Temperature 0.0 (결정적)          Temperature 1.0 (창의적)
─────────────────────          ─────────────────────
항상 같은 답변 생성               매번 다른 답변 생성
정확하고 일관됨                    다양하고 창의적
사실 기반 작업에 적합              글쓰기, 브레인스토밍에 적합

예: "1+1은?"                     예: "시를 써줘"
답: "2" (항상 동일)               답: (매번 다른 시)
```

```
   확률 분포 시각화

   Temperature = 0 (가장 높은 확률의 단어만 선택)
   ┌────────────────────────────────────────┐
   │  ████████████████████  "서울"  (95%)    │
   │  ██                    "부산"  (3%)     │
   │  █                     "대구"  (2%)     │
   └────────────────────────────────────────┘

   Temperature = 1.0 (확률 분포를 더 균등하게)
   ┌────────────────────────────────────────┐
   │  ████████████          "서울"  (50%)    │
   │  ██████                "부산"  (25%)    │
   │  █████                 "대구"  (25%)    │
   └────────────────────────────────────────┘
```

#### 주요 LLM 모델 비교

| 모델 | 제공사 | 특징 | 용도 예시 |
|------|-------|------|----------|
| Claude (Sonnet/Haiku) | Anthropic | 안전성 강조, 긴 컨텍스트 | 문서 분석, 코드 리뷰 |
| GPT-4o | OpenAI | 범용 고성능 | 복잡한 추론, 코드 생성 |
| Gemini | Google | 멀티모달 강점 | 이미지+텍스트 분석 |
| Llama | Meta | 오픈소스 | 자체 서버 배포 |

### 1.4 에이전트 아키텍처 패턴

에이전트를 설계하는 대표적인 패턴 3가지를 살펴봅니다.

#### 패턴 1: ReAct (Reasoning + Acting)

**"생각하고, 행동하고, 관찰하고, 반복한다"**

가장 널리 사용되는 패턴입니다. 사람의 사고 과정과 유사합니다.

```
┌──────────────────────────────────────────────────────┐
│                  ReAct 패턴 흐름도                      │
│                                                      │
│    ┌──────────┐                                      │
│    │  입력     │                                      │
│    │ (질문)    │                                      │
│    └────┬─────┘                                      │
│         ▼                                            │
│    ┌──────────┐    ┌──────────┐    ┌──────────┐     │
│    │ Thought  │───▶│  Action  │───▶│Observation│     │
│    │ (추론)   │    │  (행동)  │    │  (관찰)   │     │
│    └──────────┘    └──────────┘    └────┬─────┘     │
│         ▲                               │            │
│         │        아직 미완료?             │            │
│         └───────────────────────────────┘            │
│                      │                               │
│                  완료됨!                               │
│                      ▼                               │
│                ┌──────────┐                          │
│                │ 최종 응답  │                          │
│                └──────────┘                          │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**예시**: "서울 날씨에 맞는 옷 추천해줘"

| 단계 | 구분 | 내용 |
|------|------|------|
| 1 | Thought | "먼저 서울 현재 날씨를 확인해야겠다" |
| 2 | Action | weather_api("서울") 호출 |
| 3 | Observation | "서울: 22°C, 맑음" |
| 4 | Thought | "22°C 맑은 날씨에 맞는 옷을 추천하자" |
| 5 | Action | 최종 답변 생성 |
| 6 | 응답 | "가벼운 긴팔 셔츠나 얇은 가디건을 추천합니다" |

#### 패턴 2: Plan-and-Execute

**"먼저 계획을 세우고, 단계별로 실행한다"**

복잡한 작업에 적합합니다. 사람이 프로젝트를 계획하는 것과 유사합니다.

```
┌──────────────────────────────────────────────────────┐
│              Plan-and-Execute 패턴 흐름도               │
│                                                      │
│    ┌──────────┐                                      │
│    │  입력     │                                      │
│    │ (복잡한   │                                      │
│    │  작업)    │                                      │
│    └────┬─────┘                                      │
│         ▼                                            │
│    ┌──────────────────────────────┐                  │
│    │  Planner (계획 수립)          │                  │
│    │  Step 1: 데이터 수집          │                  │
│    │  Step 2: 데이터 분석          │                  │
│    │  Step 3: 보고서 작성          │                  │
│    └────┬─────────────────────────┘                  │
│         ▼                                            │
│    ┌──────────┐  ┌──────────┐  ┌──────────┐        │
│    │ Execute  │─▶│ Execute  │─▶│ Execute  │        │
│    │ Step 1   │  │ Step 2   │  │ Step 3   │        │
│    └──────────┘  └──────────┘  └────┬─────┘        │
│                                     ▼               │
│                              ┌──────────┐           │
│                              │ 최종 결과  │           │
│                              └──────────┘           │
│                                                      │
└──────────────────────────────────────────────────────┘
```

#### 패턴 3: Tool-use (도구 사용)

**"필요한 도구를 선택하여 사용한다"**

LLM이 상황에 맞는 도구를 골라 실행하는 패턴입니다.

```
┌──────────────────────────────────────────────────────┐
│               Tool-use 패턴 흐름도                      │
│                                                      │
│    ┌──────────┐                                      │
│    │  입력     │                                      │
│    └────┬─────┘                                      │
│         ▼                                            │
│    ┌──────────┐    어떤 도구를                         │
│    │   LLM    │    사용할까?                           │
│    │  (판단)  │──────────────┐                        │
│    └──────────┘              │                        │
│                              ▼                        │
│          ┌──────────────────────────────┐            │
│          │      도구 상자 (Tool Box)      │            │
│          │                              │            │
│          │  🔍 검색    📧 이메일         │            │
│          │  📊 분석    💾 저장           │            │
│          │  🌐 API    📝 문서 작성       │            │
│          │                              │            │
│          └──────────────────────────────┘            │
│                              │                        │
│                              ▼                        │
│                       ┌──────────┐                   │
│                       │ 도구 실행  │                   │
│                       │ 결과 반환  │                   │
│                       └──────────┘                   │
│                                                      │
└──────────────────────────────────────────────────────┘
```

#### 패턴 비교 요약

| 패턴 | 장점 | 단점 | 적합한 경우 |
|------|------|------|-----------|
| **ReAct** | 유연, 적응적 | 비효율적일 수 있음 | 탐색적 작업, 대화형 |
| **Plan-and-Execute** | 체계적, 예측 가능 | 계획 수정이 어려움 | 복잡하고 정형화된 작업 |
| **Tool-use** | 확장성, 모듈화 | 도구 선택 실수 가능 | 다양한 도구가 필요한 경우 |

> **실무에서는** 이 패턴들을 조합하여 사용합니다.
> 예를 들어, ReAct + Tool-use 패턴이 가장 일반적입니다.

### 1.5 에이전트 구성 요소

LangGraph에서 에이전트는 다음 5가지 요소로 구성됩니다:

```
┌──────────────────────────────────────────────────────────┐
│                  에이전트 구성 요소 전체 그림                 │
│                                                          │
│   ┌────────────────────────────────────────────┐        │
│   │              그래프 (Graph)                  │        │
│   │                                            │        │
│   │   ┌────────────────────────────────────┐   │        │
│   │   │          상태 (State)               │   │        │
│   │   │   messages: [...]                  │   │        │
│   │   │   current_step: "분석중"            │   │        │
│   │   │   result: null                     │   │        │
│   │   └────────────────────────────────────┘   │        │
│   │                                            │        │
│   │   ┌──────┐  엣지(Edge)  ┌──────┐          │        │
│   │   │노드 A │────────────▶│노드 B │          │        │
│   │   │(분석) │             │(생성) │          │        │
│   │   └──────┘              └──────┘          │        │
│   │       │                     │              │        │
│   │       ▼                     ▼              │        │
│   │   ┌──────┐              ┌──────┐          │        │
│   │   │도구 1 │              │도구 2 │          │        │
│   │   │(검색) │              │(저장) │          │        │
│   │   └──────┘              └──────┘          │        │
│   │                                            │        │
│   └────────────────────────────────────────────┘        │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### (1) 노드 (Node)

> **노드**는 에이전트가 수행하는 **개별 작업 단위**입니다.

사람으로 비유하면 "하나의 업무 단계"에 해당합니다.

| 노드 예시 | 설명 |
|----------|------|
| 분석 노드 | 입력 데이터를 분석하는 단계 |
| LLM 호출 노드 | AI 모델에게 질문하는 단계 |
| 도구 실행 노드 | 외부 API를 호출하는 단계 |
| 결과 정리 노드 | 결과를 포맷팅하는 단계 |

#### (2) 엣지 (Edge)

> **엣지**는 노드와 노드를 연결하는 **흐름의 방향**입니다.

두 가지 종류가 있습니다:

```
일반 엣지 (항상 같은 방향):
   노드 A ──────────▶ 노드 B
   (무조건 A 다음에 B 실행)

조건부 엣지 (조건에 따라 분기):
                    ┌──▶ 노드 B (조건 1)
   노드 A ──(판단)──┤
                    └──▶ 노드 C (조건 2)
   (A의 결과에 따라 B 또는 C 실행)
```

#### (3) 상태 (State)

> **상태**는 에이전트가 작업하면서 **공유하는 데이터 공간**입니다.

사무실의 공유 화이트보드와 같습니다. 모든 노드가 상태를 읽고 업데이트합니다.

```
┌─────────────── 상태 (State) ───────────────┐
│                                            │
│   messages: ["안녕하세요", "무엇을 도와..."]  │
│   user_intent: "날씨 질문"                   │
│   tool_results: {"temp": 22}               │
│   final_answer: null                       │
│                                            │
│   ← 노드 A가 messages를 읽음                │
│   ← 노드 B가 user_intent를 업데이트          │
│   ← 노드 C가 tool_results를 업데이트         │
│   ← 노드 D가 final_answer를 작성            │
│                                            │
└────────────────────────────────────────────┘
```

#### (4) 그래프 (Graph)

> **그래프**는 노드와 엣지를 연결한 **전체 워크플로우**입니다.

에이전트의 전체 로직을 하나의 그래프로 표현합니다.

```
   그래프 = 노드 + 엣지 + 상태의 조합

   ┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐
   │START │───▶│분석  │───▶│생성  │───▶│ END │
   └─────┘    └─────┘    └─────┘    └─────┘
```

#### (5) 도구 (Tool)

> **도구**는 에이전트가 외부 세계와 **상호작용하는 수단**입니다.

사람이 컴퓨터, 전화, 이메일을 사용하듯, 에이전트도 도구를 사용합니다.

| 도구 종류 | 예시 |
|----------|------|
| 검색 도구 | 웹 검색, 데이터베이스 조회 |
| 계산 도구 | 수학 계산, 통계 분석 |
| 통신 도구 | 이메일 발송, 슬랙 메시지 |
| 파일 도구 | 파일 읽기/쓰기, PDF 생성 |
| API 도구 | 외부 서비스 호출 (날씨, 지도 등) |

#### 비즈니스/일반

## 2. 단계별 실습

이 모듈에서는 코드를 작성하지 않습니다. 대신 에이전트의 동작을 **직접 설계해보는**
사고 실습을 진행합니다.

### 실습 1: 에이전트 동작 흐름 따라가기

다음 시나리오를 읽고, 에이전트가 어떤 순서로 동작할지 생각해보세요.

**시나리오**: 사용자가 "오늘 서울 날씨를 확인하고, 비가 오면 실내 데이트 코스를 추천해줘"
라고 입력했습니다.

**에이전트가 사용할 수 있는 도구**:
- `get_weather(city)` - 도시의 현재 날씨 조회
- `search_places(query, category)` - 장소 검색
- `get_reviews(place_id)` - 장소 리뷰 조회

**풀어볼 질문**:
1. 에이전트의 첫 번째 Thought(추론)는 무엇일까요?
2. 몇 번의 도구 호출이 필요할까요?
3. 날씨가 맑다면 에이전트의 행동이 어떻게 달라질까요?

<details>
<summary>정답 예시 (클릭하여 펼치기)</summary>

```
Step 1 - Thought: "먼저 서울 날씨를 확인해야 한다"
Step 2 - Action:  get_weather("서울")
Step 3 - Observation: "서울: 비, 15°C"
Step 4 - Thought: "비가 오니까 실내 데이트 코스를 검색해야 한다"
Step 5 - Action:  search_places("서울 실내 데이트", "leisure")
Step 6 - Observation: ["미술관 A", "카페 B", "영화관 C"]
Step 7 - Thought: "추천할 장소의 리뷰를 확인하자"
Step 8 - Action:  get_reviews("미술관 A")
Step 9 - Observation: "평점 4.5, 분위기 좋다는 평이 많음"
Step 10 - 최종 응답: "오늘 서울은 비가 오고 15°C입니다.
          실내 데이트 코스로 미술관 A를 추천합니다. (평점 4.5)..."

→ 도구 호출: 총 3번 (날씨 1번, 장소 검색 1번, 리뷰 1번)
→ 날씨가 맑다면: "야외 데이트 코스"로 검색이 변경됨
```

</details>

## 3. 실전 예제

### 예제: "고객 문의 처리 에이전트" 설계

실제 비즈니스에서 사용할 수 있는 에이전트를 개념적으로 설계해봅시다.

#### 요구사항

- 고객의 문의 이메일을 자동으로 처리
- 문의 유형을 분류 (환불, 기술 지원, 일반 문의, 불만)
- 유형에 따라 적절한 부서로 전달
- 간단한 문의는 자동 응답

#### 설계 (그래프 구조)

```
┌──────────────────────────────────────────────────────────┐
│           고객 문의 처리 에이전트 설계                        │
│                                                          │
│   ┌─────────┐                                            │
│   │  START   │                                            │
│   │ (문의수신)│                                            │
│   └────┬────┘                                            │
│        ▼                                                 │
│   ┌─────────┐                                            │
│   │ 문의 분류 │  ← LLM이 문의 내용을 분석하여 유형 판단       │
│   │  (노드1) │                                            │
│   └────┬────┘                                            │
│        │                                                 │
│        ├── 환불 요청 ──────▶ ┌─────────┐                  │
│        │                    │ 환불 처리  │ ← 환불 정책 확인   │
│        │                    │  (노드2a) │   & 자동 처리     │
│        │                    └─────────┘                  │
│        │                                                 │
│        ├── 기술 지원 ──────▶ ┌─────────┐                  │
│        │                    │ 기술 지원  │ ← FAQ 검색       │
│        │                    │  (노드2b) │   & 해결책 제시   │
│        │                    └─────────┘                  │
│        │                                                 │
│        ├── 일반 문의 ──────▶ ┌─────────┐                  │
│        │                    │ 자동 응답  │ ← 템플릿 기반     │
│        │                    │  (노드2c) │   응답 생성       │
│        │                    └─────────┘                  │
│        │                                                 │
│        └── 불만 접수 ──────▶ ┌─────────┐                  │
│                             │ 담당자    │ ← 사람에게        │
│                             │ 에스컬레이션│  즉시 전달       │
│                             │  (노드2d) │                  │
│                             └─────────┘                  │
│                                   │                      │
│                                   ▼                      │
│                             ┌─────────┐                  │
│                             │ 결과 기록 │ ← DB에 처리 결과   │
│                             │  (노드3) │   저장            │
│                             └────┬────┘                  │
│                                  ▼                       │
│                             ┌─────────┐                  │
│                             │   END   │                  │
│                             └─────────┘                  │
│                                                          │
│   [State 구성]                                            │
│   - email_content: 원본 이메일 내용                        │
│   - category: 분류 결과 (환불/기술/일반/불만)                │
│   - response: 생성된 응답                                  │
│   - escalated: 에스컬레이션 여부                            │
│                                                          │
│   [사용 도구]                                              │
│   - 이메일 API: 이메일 수신/발신                            │
│   - DB API: 고객 정보 조회, 처리 결과 저장                   │
│   - FAQ 검색: 기술 문서 검색                                │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 구성 요소 정리

| 구성 요소 | 이 에이전트에서의 역할 |
|----------|---------------------|
| **노드** | 문의 분류, 환불 처리, 기술 지원, 자동 응답, 에스컬레이션, 결과 기록 |
| **엣지** | 노드 간 연결 (분류 결과에 따른 조건부 엣지 포함) |
| **상태** | email_content, category, response, escalated |
| **그래프** | 위의 전체 흐름도 |
| **도구** | 이메일 API, DB API, FAQ 검색 |

4. Thought(추론) → Action(행동) → Observation(관찰)
5. 도구(Tool)

</details>

### 문제 2: 에이전트 설계

다음 시나리오에서 에이전트를 설계해보세요. (종이에 그려도 좋습니다)

**시나리오**: "회의록 정리 에이전트"
- 회의 녹음을 텍스트로 변환 (음성 → 텍스트)
- 핵심 내용 요약
- 액션 아이템(할 일) 추출
- 참석자별 발언 정리
- 결과를 슬랙으로 전송

**설계해볼 것**:
1. 필요한 노드 목록
2. 노드 간 연결 (엣지) 순서
3. 상태에 포함될 데이터
4. 필요한 도구 목록

<details>
<summary>정답 예시</summary>

**노드**: (1) 음성→텍스트 변환, (2) 내용 요약 (LLM), (3) 액션 아이템 추출 (LLM),
        (4) 참석자별 분류, (5) 슬랙 전송

**엣지**: START → (1) → (2) → (3) → (4) → (5) → END

**상태**: raw_audio, transcript, summary, action_items, speaker_segments, slack_result

**도구**: 음성인식 API(Whisper 등), 슬랙 API, 캘린더 API(참석자 확인)

</details>

### 문제 3: 적합성 판단

다음 각 상황에서 AI 에이전트가 적합한지(O) 부적합한지(X) 판단하고 이유를 적어보세요:

1. ( ) 매일 오전 9시에 같은 형식의 보고서를 생성하는 작업
2. ( ) 다양한 형식의 이력서를 분석하여 핵심 역량을 추출하는 작업
3. ( ) 주식 거래를 밀리초 단위로 실행하는 고빈도 트레이딩
4. ( ) 고객의 자유 형식 피드백을 분류하고 요약하는 작업
5. ( ) CSV 파일의 A열과 B열을 합산하는 작업

<details>
<summary>정답</summary>

4. (O) - 자연어 처리, 분류 판단 필요, 에이전트에 적합
